# Markdown Structure Contracts

This document defines the strict section ordering and structural contracts for `proposal.md`, `standards.md`, `specs/*.md`, and `decisions/*.md` files in RuneContext.

## proposal.md Structure

`proposal.md` is the canonical reviewable intent artifact for a change. It must use a strict structure to enable reliable parsing by tooling and RuneCode.

### Required Sections

The following sections must appear in **exact order**, using level-2 markdown headings (`##`):

1. **Summary**
   - Concise one-paragraph description of the change.
   - Audience: anyone reading the change for the first time.
   - Required; may contain `N/A` only if explicitly justified inline.

2. **Problem**
   - What problem, gap, or opportunity does this change address?
   - Why does it matter?
   - Required; may contain `N/A` only if this is a pure refactoring or chore with no problem context.

3. **Proposed Change**
   - What is the concrete proposal?
   - What is being added, modified, or removed?
   - Describe the solution at a level of detail sufficient for design review.
   - Required; may not be `N/A`.

4. **Why Now**
   - Why is this change urgent or important at this moment?
   - What triggers or enables it?
   - Required; may contain `N/A` only if timing is not a factor.

5. **Assumptions**
   - What assumptions are being made about the codebase, platform, requirements, or user behavior?
   - What inferred defaults did tooling apply (e.g., likely context bundle, verification method)?
   - Required; if no assumptions, state `N/A` explicitly.

6. **Out of Scope**
   - What is explicitly NOT being addressed in this change?
   - What is deferred to future work?
   - Required; may contain `N/A` if nothing is deferred.

7. **Impact**
   - User-facing impact (if any).
   - Performance, security, or operational impact.
   - Migration or rollout considerations.
   - Required; may contain `N/A` only if there is genuinely no impact.

### Additional Sections

Additional custom sections may appear **after** the required section block. They are optional and may be used for:
- Detailed design discussion
- Alternative approaches considered
- Known limitations or future improvements
- References and links

### Parsing and Validation Rules

- **Section identification**: Tooling must identify sections by their level-2 headings (`## Section Name`).
- **Ordering enforcement**: Required sections must appear in the exact order listed above.
- **Content validation**: Each required section must either:
  - Contain substantive content (one or more paragraphs, lists, code blocks, etc.), OR
  - Contain the text `N/A` on its own line (indicating the section does not apply)
- **No duplication**: Required sections must not appear more than once.
- **No section nesting under required sections**: Content between the current section header and the next section header is part of the current section. Level-2 headings at or above the current level mark the next section.

### Example Structure

```markdown
## Summary
One-sentence or one-paragraph overview.

## Problem
Description of the problem.

## Proposed Change
Description of the proposal.

## Why Now
Justification for timing.

## Assumptions
Key assumptions and inferred defaults.

## Out of Scope
What is not being addressed.

## Impact
Impact assessment.

## Additional Context (Optional)
Any additional detail beyond the required sections.
```

## standards.md Structure

`standards.md` must always be present in a change folder. It is automatically maintained by tooling but remains canonical and user-reviewable.

### Recommended Sections

1. **Applicable Standards**
   - Normalized list of standards that apply to this change.
   - Each standard listed with its path and short description.
   - This section may be regenerated by tooling but remains reviewable.
   - Example:
     ```markdown
     ## Applicable Standards
     - `standards/global/deterministic-check-write.md`: ...
     - `standards/security/trust-boundary-interfaces.md`: ...
     ```

2. **Standards Added Since Last Refresh** (Optional)
   - Standards that were newly discovered or added since the last change shape/refresh.
   - Used to highlight what changed during the latest refresh.
   - Example:
     ```markdown
     ## Standards Added Since Last Refresh
     - `standards/backend/api-versioning.md`: New dependency discovered during implementation planning.
     ```

3. **Standards Considered But Excluded** (Optional)
   - Standards that were evaluated but determined not to apply.
   - Used to document decisions during standards discovery.
   - Example:
     ```markdown
     ## Standards Considered But Excluded
     - `standards/frontend/accessibility.md`: Not applicable; backend-only change.
     ```

4. **Resolution Notes** (Optional)
   - Additional notes about standards selection, conflicts, or special considerations.
   - Example:
     ```markdown
     ## Resolution Notes
     This change touches standards/security/trust-boundary-interfaces.md, which conflicts with...
     ```

### Maintenance Rules

- **Automatic refresh**: Tooling may refresh `standards.md` automatically when:
  - A change is created (`change new`)
  - A change is shaped/reshaped (`change shape`)
  - RuneCode discovers new relevant standards during planning
- **Reviewable diffs**: Any automatic refresh must produce a reviewable diff before commit/merge. Tooling must not silently rewrite the file.
- **User-editable**: Users may still edit `standards.md` manually when needed.
- **Always present**: `standards.md` must exist and be non-empty in every change folder.

### Canonical Nature

- `standards.md` is a **reviewable canonical file**, not a disposable generated cache.
- Changes to `standards.md` should be reviewed as part of the larger change diff.
- The file remains part of the permanent change history.

## Special Handling For Custom Markdown

RuneContext supports minimal custom/optional markdown files such as `design.md`, `tasks.md`, `references.md`, and `verification.md`. These files have no mandated structure in v1 and are authored freely by users or tooling. However:

- They must use standard markdown syntax.
- They are not parsed by RuneContext core; they are advisory documentation.
- If RuneCode or other tools later create structured contracts for these files, they will be published in a future version of this document.

## specs/*.md Frontmatter Contract

Specs remain hand-authored markdown documents, but they now carry a strict YAML frontmatter block for traceability metadata.

### Required Frontmatter Fields

The YAML frontmatter for `specs/*.md` must validate against `schemas/spec.schema.json` and include:

- `schema_version`
  - required and fixed to `1`
- `id`
  - required stable spec identifier
  - must match the path-relative stem under `specs/` without the `.md` extension
  - example: `runecontext/specs/auth-gateway.md` -> `auth-gateway`
- `title`
  - required human-readable spec title
- `originating_changes`
  - required non-empty array of change IDs that originally promoted the spec into durable knowledge
- `revised_by_changes`
  - required array of later change IDs that revised the spec

### Rules

- The frontmatter block must appear at the start of the file.
- The body after frontmatter remains freeform markdown in v1.
- `id` path matching is strict and fail-closed.
- Change references in `originating_changes` and `revised_by_changes` must resolve to real change folders during project-level validation.
- Specs should reference standards by path in the markdown body when needed; they should not copy entire standards into the spec body.

## decisions/*.md Frontmatter Contract

Decisions remain hand-authored markdown documents, but they now carry a strict YAML frontmatter block for traceability metadata.

### Required Frontmatter Fields

The YAML frontmatter for `decisions/*.md` must validate against `schemas/decision.schema.json` and include:

- `schema_version`
  - required and fixed to `1`
- `id`
  - required stable decision identifier
  - must match the path-relative stem under `decisions/` without the `.md` extension
  - example: `runecontext/decisions/DEC-0001-trust-boundary-model.md` -> `DEC-0001-trust-boundary-model`
- `title`
  - required human-readable decision title
- `originating_changes`
  - required non-empty array of change IDs that originally promoted the decision
- `related_changes`
  - required array of later change IDs that materially relate to, rely on, or revisit the decision

### Rules

- The frontmatter block must appear at the start of the file.
- The body after frontmatter remains freeform markdown in v1.
- `id` path matching is strict and fail-closed.
- Change references in `originating_changes` and `related_changes` must resolve to real change folders during project-level validation.
- Decisions are durable ADR-like references; richer lifecycle/state metadata is intentionally deferred beyond alpha.1.
